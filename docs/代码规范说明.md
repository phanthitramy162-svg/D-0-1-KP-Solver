# D{0-1}KP 智能求解系统 - 项目编程规范

本项目在开发过程中，参考了《阿里巴巴 Java 开发手册》与《腾讯 C++ 代码规范》的工程化思想，并结合 Python 官方的 PEP 8 风格指南，制定了以下严谨的编码规范，并在 `src` 目录下的所有源码中严格执行。

## 1. 语句缩进
* **规则**：统一使用 **4 个空格** 进行缩进。
* **强制**：严禁使用 Tab 键，绝对禁止空格与 Tab 混用，以保证在不同编辑器和操作系统下代码格式的一致性。

## 2. 变量命名
* **规则**：采用全小写字母，单词之间使用下划线分隔（Snake Case / 蛇形命名法）。
* **要求**：命名必须具备明确的业务含义，做到“见名知意”。禁止使用无意义的拼音缩写或单字母（除循环控制变量 `i`, `j`, `k` 外）。
* **示例**：`max_profit`, `sorted_items`, `time_cost`。

## 3. 函数与类命名
* **类命名**：采用大驼峰命名法（UpperCamelCase / PascalCase），名词优先。
  * **示例**：`DKPGUI`
* **函数命名**：采用全小写字母与下划线分隔（Snake Case），且优先使用“动宾结构”以表达函数动作。
  * **示例**：`read_all_kp_instances()`, `solve_dp()`, `plot_comparison()`。

## 4. 常量命名
* **规则**：常量命名必须全部大写，单词之间用下划线分隔。
* **示例**：`MAX_CAPACITY`, `DEFAULT_FILE_PATH`。

## 5. 每行最多字符数
* **规则**：单行代码字符数上限严格控制在 **120 个字符**以内。
* **处理方式**：若代码过长，需使用括号 `()` 实现隐式换行，或使用反斜杠 `\` 进行显式换行，换行后需保持适当缩进以对齐逻辑。

## 6. 函数最大行数
* **规则**：遵循“单一职责原则（高内聚低耦合）”，单个函数体的有效代码行数**不应超过 80 行**。
* **处理方式**：若函数逻辑过长（如上帝函数），必须将其拆分为多个功能单一的私有子函数。

## 7. 空行规则
* **模块级空行**：顶级函数定义和类定义之间，保留 **2 个空行**。
* **类内部空行**：类内部的方法定义之间，保留 **1 个空行**。
* **逻辑块空行**：在同一个函数体内部，不同的业务逻辑块（如变量初始化区、核心循环区、结果返回区）之间可使用 1 个空行进行视觉隔离。

## 8. 注释规则
* **文档注释 (Docstrings)**：所有的公共类和核心函数正下方，必须使用三引号 `"""` 编写清晰的描述，包含功能说明、参数释义和返回值含义。
* **单行注释**：使用 `#`。`#` 与注释文字之间必须**保留一个空格**（如 `# 这是一个注释`）。若是行尾注释，`#` 必须距离前面的代码至少两个空格。
* **原则**：代码本身应具有良好的自解释性，注释应解释“为什么这么做（Why）”，而不是仅仅翻译“代码在做什么（What）”。

## 9. 操作符前后空格
* **二元操作符**：赋值（`=`）、比较（`==`, `<`, `>`, `!=`）、布尔（`and`, `or`, `not`）和二元算术操作符（`+`, `-`, `*`, `/`）的两侧，必须**保留一个空格**。
  * **正例**：`a = b + c`
  * **反例**：`a=b+c`
* **例外**：函数定义或调用时，关键字参数的 `=` 号两侧禁止加空格（如 `plt.figure(figsize=(10, 6))`）。

## 10. 其他规则
* **Imports 导入规范**：导入语句必须放在文件顶部。按顺序分为三组：标准库导入、第三方依赖库导入、本项目内部模块导入。每组之间保留一个空行。
* **魔法数字 (Magic Numbers)**：代码中严禁直接出现无解释的硬编码数字（除 0, 1, -1 外），应当定义为具有明确含义的常量或变量。
